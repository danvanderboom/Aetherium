## Why

Four `wire-X-live` slices (death/respawn, abilities, progression, factions) made gameplay rules per-world **data** — `DeathPolicy`/`AbilityConfig`/`ProgressionConfig`/`FactionConfig`, threaded through `CreateWorldRequest`/`WorldConfig`/`WorldTemplate` and proven end-to-end by grain tests. But a survey (2026-07-10) confirmed the plumbing has no data source: **the only code that ever populates these configs is test code.** The REST path (`CreateWorldRequestDto`) has no fields for them, so every production world runs with null configs. The engine's core promise — "many different games from the same engine, defined declaratively" — has working pipes and no faucet.

This change adds the faucet: **game definition bundles** — YAML directories under `Data/Games/` that each define a complete game (world shape + all gameplay-rule configs) — plus a registry that loads and validates them, and an instance-creation path that turns a definition into a running, playable world. Multiple definitions coexist on one server, each with multiple concurrently-running instances: three instances of a fantasy RPG beside two instances of a sci-fi hacking game, all on the same silo, all defined purely in YAML.

## What Changes

- **`GameDefinition` model** (`Aetherium.Model.Games`): id, name, semver version, description, tags; a `world` section (generator, size, max players, narrative); and the four shipped per-world configs (`DeathPolicy?`, `AbilityConfig?`, `ProgressionConfig?`, `FactionConfig?`). Sections the engine hasn't shipped yet (economy, party, events, telemetry, locale packs) are **not** in v1 — the schema grows as each subsystem's config type ships, same seam, no redesign.
- **Bundle loader** (`Aetherium.Server.Games.GameDefinitionLoader`): a bundle is a directory containing `game.yaml`; large sections may split into conventional sibling files (`abilities.yaml`, `progression.yaml`, `factions.yaml`, `death.yaml`) — each section from exactly one place, duplicates rejected. YAML 1.2 via YamlDotNet (new dependency, `Aetherium.Server` only), camelCase keys binding to the existing config POCOs. Malformed bundles are rejected with actionable diagnostics and never block other bundles or server startup.
- **Validation** (`GameDefinitionValidator`): structural strictness (unknown keys are errors — typos must not silently do nothing) plus cross-section referential checks (a skill's `unlocksAbilityId` must exist in `abilities`; XP-award rules must reference declared pools; faction relations must reference declared factions; …).
- **Registry** (`GameDefinitionRegistry`, DI singleton in the `PrefabLibrary`/`HubWorldLoader` mold): loads `Data/Games/**` at startup, exposes list/get, rejects duplicate game ids.
- **Instance path**: `IGameManagementGrain.CreateGameInstanceAsync(gameId, instanceName?)` maps a definition to a `CreateWorldRequest` (all four configs populated) through the **existing** world-creation machinery; `WorldConfig`/`WorldInfo` gain `GameDefinitionId`/`GameDefinitionVersion` so `ListGameInstancesAsync(gameId)` can enumerate a game's running instances. Joining/playing an instance uses the existing world-join surface unchanged.
- **Instance immutability**: configs are already copied into world/map grain state at creation, so a running instance is immune to bundle edits/reloads by construction; the instance records which definition version built it.
- **Two sample bundles** shipped under `Data/Games/` — `emberfall` (fantasy RPG: mana pools, spells, town/cult factions, XP progression, respawn-at-town death) and `neonveil` (sci-fi netrunning: bandwidth pool, breach/spike programs, corp/collective factions, harsher death policy) — serving as living documentation and as the fixtures for the concurrent multi-game integration tests.
- **Management surface**: REST endpoints to list definitions and create/list instances; `aetherctl game list|instances|create` commands.

## Impact

- Affected specs: new capability `game-definitions`.
- Affected code: new `Aetherium.Model/Games/GameDefinition.cs`; new `Aetherium.Server/Games/{GameDefinitionLoader,GameDefinitionValidator,GameDefinitionRegistry,GameDefinitionMapper}.cs`; `WorldModels.cs`/`WorldContracts.cs` gain definition-id fields (additive `[Id(n)]`); `GameManagementGrain` + `IGameManagementGrain` gain the instance methods; `ManagementController` + `aetherctl` surface; `Data/Games/emberfall/`, `Data/Games/neonveil/`; new tests under `Aetherium.Test/Games/`. New NuGet dependency: YamlDotNet (server only).
- Non-breaking: all fields additive/nullable; existing world-creation paths untouched; worlds created without a definition behave exactly as today.
