## Why

`add-character-progression` (Wave 1 §4.4) shipped `ProgressPool`/`ILevelCurve`/`Attributes`/`SkillDefinition`/`SkillCatalog`/`UnlockedSkills`/`SkillUnlockService`/`RoleAffinity` as pure, unreferenced primitives — confirmed by grep, nothing outside `Aetherium.Server/Progression/` and its own tests touches them. No entity carries any of them. XP is never awarded, no skill is ever unlocked in gameplay, and `SkillDefinition.UnlocksAbilityId` resolves against nothing.

Progression is the connective tissue of Wave 1: it turns the kills that combat/abilities already produce into XP, XP into levels, levels into skills, and skills into ability grants and attribute changes. Wiring it live makes the systems we already shipped feed each other. Per the same vision that shaped `wire-death-respawn-live` and `wire-abilities-live`, it must be **per-world declarative data** — a fantasy game's "Strength → Warrior skills → Fireball" and a sci-fi game's "Hacking → Netrunner skills → ICE-breaker" are the same machinery with different `ProgressionConfig` rows. The data tier is authored as YAML later (no downstream change) and is the natural seam for ECA-style award/skill rules.

## What Changes

- **New data tier (`Aetherium.Model.Progression`):** `ProgressionConfig` bundling `ProgressPoolDefinition`s (each naming a `LevelCurveDefinition`), a skill catalog of `SkillDefinitionData`, `StartingAttributes`/`StartingRoleAffinity`, declarative `XpAwardRule`s, and `AttributeDerivation`s (e.g. vitality → max health). All `[GenerateSerializer]`, reachable from `WorldConfig` and `WorldTemplate` like `DeathPolicy`/`AbilityConfig`. The behavior-bearing primitives (`ILevelCurve`, `SkillCatalog`) stay in `Server`; a `ProgressionCompiler` bridges data → runtime (the ContentAtlas data/seeding split, same as `AbilityCompiler`).
- **Per-world threading:** `ProgressionConfig` rides `WorldConfig`/`WorldTemplate`/`CreateWorldRequest` → `WorldGrainState` → `AddMapAsync` → `IGameMapGrain.InitializeAsync` → persisted on `MapState`, rehydrated in `OnActivateAsync`. Engine ships no progression content.
- **Character stamping:** `JoinPlayerAsync` stamps a joining character with fresh `ProgressPools`/`Attributes`/`UnlockedSkills`/`RoleAffinity` from the world's config (nothing when the world declares none — `Character`'s constructor is unchanged, staying genre-neutral).
- **The XP loop:** at the shared monster-defeat chokepoint (both `AttackAsync` and `UseAbilityAsync`), a declarative `XpAwardRule` set awards XP to the killer's pools and recomputes levels — one helper feeding both melee and ability kills so they never diverge.
- **Skill unlock + ability-grant bridge:** new `IGameMapGrain.UnlockSkillAsync(sessionId, skillId)` runs `SkillUnlockService` (prerequisite-gated, plus an optional per-skill `RequiredPoolLevel` so XP actually gates skills), then applies the skill's effects: `ModifiesAttributeId` adjusts `Attributes` (re-deriving stats), and `UnlocksAbilityId` adds to a new per-player `GrantedAbilities` component. A per-world `RequireSkillToCastAbilities` flag (default **false**, preserving today's "any catalog ability is castable") makes `GrantedAbilities` an actual cast gate when a game opts in.
- **Attribute coupling:** declarative `AttributeDerivation`s map attributes onto derived stats (vitality → `Health.MaxLevel`, speed → `ActionSpeed.Speed`), applied on-change (join, skill-modify) rather than per-tick.
- **Observability:** `GetProgressionAsync(sessionId)` read accessor (pools/levels/xp, attributes, unlocked skills, granted abilities).

## Impact

- Affected specs: `character-progression` (new live-wiring requirements), `abilities` (optional skill-gated casting), `engine-core` (Character stamped with progression at join).
- Affected code: `Aetherium.Model/Progression/*` (new), `Aetherium.Server/Progression/*` (new compiler + `GrantedAbilities`; `ProgressPool`/skill types gain nothing structural), `Aetherium.Server/MultiWorld/{GameMapGrain,IGameMapGrain,WorldGrain,WorldModels}.cs`, `Aetherium.Model/Worlds/WorldContracts.cs`, `Aetherium.Server/Management/GameManagementGrain.cs`, `Aetherium.Server/Services/OrleansWorldHost.cs`.
- Explicitly deferred (see design.md Non-Goals): meta-progression handoff (`MetaProgressionGrain`); XP from exploration/quests (kills only this slice); a skill-point economy (prerequisite + optional level gate only); full ECA generalization of award/skill rules; cross-session/reactivation persistence of *earned* progression (ephemeral within-run, like player sessions and resource pools today); monster progression.
