## Phase E — Compaction and retention

- [x] E.1 [GameMapGrain](Aetherium.Server/MultiWorld/GameMapGrain.cs) registers a periodic `RegisterGrainTimer` in `OnActivateAsync` (and disposes it in `OnDeactivateAsync`). On fire, the callback checks whether any deltas have been appended since the last snapshot and, if so, awaits `ForceSnapshotAsync` then resets the counter. Periodic timer interval comes from `CompactionOptions.IntervalMinutes` (default 10)
- [x] E.2 Threshold-based trigger: `PersistDeltaAsync` increments `_deltasSinceLastSnapshot` after every successful append and immediately fires a fire-and-forget compaction when the counter crosses `CompactionOptions.DeltaCountThreshold` (default 1000). Both triggers share a CAS-guarded `_compactionInFlight` flag so a timer firing concurrently with a threshold cross doesn't double-fire. `ForceSnapshotAsync` already handles "capture snapshot + persist + truncate log ≤ sequence"
- [x] E.3 [Aetherium.Server/Persistence/PersistenceOptions.cs](Aetherium.Server/Persistence/PersistenceOptions.cs) — new `PersistenceOptions { Compaction { Enabled, IntervalMinutes, DeltaCountThreshold } }`. Bound from the `Persistence` section of `appsettings.json` in [Program.cs](Aetherium.Server/Program.cs). `Compaction.Enabled = false` short-circuits both the timer and the threshold trigger
- [x] E.4 [Aetherium.Test/Persistence/CompactionTests.cs](Aetherium.Test/Persistence/CompactionTests.cs) — three TestCluster integration tests configured with `DeltaCountThreshold=3` and `IntervalMinutes=60` so the threshold path is deterministic: (a) crossing the threshold drives an automatic snapshot into the store, (b) the retained log holds only entries with `Sequence > snapshot.LastSequence` after compaction, (c) the post-compaction counter resets so a single follow-up delta does NOT immediately produce another snapshot, but five do. Full suite: 788 tests pass, 0 failures, 2 skipped (was 785; +3 from this phase)

## Phase F — Schema and delta-type evolution

- [x] F.1 Added `delta_type_version INTEGER NOT NULL DEFAULT 1` to the `map_delta_log` schema in [SqliteWorldSnapshotStore](Aetherium.Server/Persistence/SqliteWorldSnapshotStore.cs). Includes a best-effort `ALTER TABLE` upgrade path so existing databases (from earlier commits) gain the column on next open
- [x] F.2 Added `virtual int DeltaTypeVersion => 1` to [MapDelta](Aetherium.Server/MultiWorld/Deltas.cs). Not serialized on the wire — intrinsic to each subtype. `AppendMapDeltaAsync` writes the current binary's version into the new column; `GetMapDeltasSinceSequenceAsync` reads the stored version, deserializes the delta, and compares stored vs. the deserialized type's `DeltaTypeVersion`. If `stored > binary` → throws new [PersistenceVersionMismatchException](Aetherium.Server/Persistence/PersistenceVersionMismatchException.cs) with the offending sequence + type. Stored ≤ binary always loads (Orleans `[Id]` handles old-field forward-compat)
- [x] F.3 Added a wire-stability contract comment block at the top of [Deltas.cs](Aetherium.Server/MultiWorld/Deltas.cs) documenting the four rules: never renumber `[Id]`, additive fields pick the next number, retired numbers stay reserved, and semantic changes bump `DeltaTypeVersion`
- [x] F.4 Added a snapshot version guard in [GameMapGrain.TryHydrateFromSnapshotAsync](Aetherium.Server/MultiWorld/GameMapGrain.cs): reads `WorldSnapshot.SnapshotVersion` after deserializing the blob, compares against `SupportedSnapshotVersion` constant (currently 1), throws `PersistenceVersionMismatchException` if higher. Dedicated test for the snapshot-version path deferred — exercising the throw through OnActivateAsync requires either silo restart or making the helper visible to tests, and the guard logic is small (three lines) and structurally identical to the delta version path which IS tested
- [x] F.5 [Aetherium.Test/Persistence/PersistenceVersioningTests.cs](Aetherium.Test/Persistence/PersistenceVersioningTests.cs) — four tests: (a) stored row with `delta_type_version=99` triggers `PersistenceVersionMismatchException` on read with type+version in the message, (b) stored version equal to binary's loads cleanly, (c) stored version below binary's loads cleanly (forward-compat upgrade scenario), (d) subclass override is observable via the virtual property. Full suite: 792 tests pass, 0 failures, 2 skipped (was 788; +4 from this phase)
